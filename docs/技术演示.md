# Mimir 技术演示叙述（评审讲解用）

> 把 Mimir 的优化串成一条「问题 → 方案 → 引擎级证据」的连贯故事，供评审讲解。
> 所有数据真实测量，Qwen3-4B-Instruct-2507，单卡 RTX 3090，vLLM 0.10.2。

## 一、问题：智能体推理的内存特征

智能体（规划→执行→反思，多轮 + 工具 + 分支）推理在内存上与线性生成显著不同：
1. **KV Cache 持续累积**——多轮不重置，显存随轮次线性增长。
2. **上下文高度冗余**——system prompt / 工具描述逐轮重复。
3. **推理路径分支**——Tree-of-Thought 多候选，分支导致 KV 指数膨胀。
4. **工具中间数据**——大 JSON / 搜索结果全量进入 KV。
5. **生命周期长**——单任务可执行数十~数百步，资源长期占用。

资源受限 / 多任务并发下，低效内存管理 → 显存暴涨 → 延迟上升甚至 OOM 任务失败。

## 二、方案：两层优化 + vLLM 内核级 patch

Mimir 把优化分两层：

**外部层（请求侧变换，`mimir/`）**——在喂给 vLLM 前改写请求：
- 上下文压缩（`mimir.context`）：摘要旧轮 + 精简工具描述。
- 工具数据外置（`mimir.tools`）：大返回外存，KV 仅留引用 + 摘要。
- 分层存储（`mimir.tiered`）：GPU/HOST/DISK 三层冷热分离。
- 分支 CoW 记账（`mimir.branch`）：分支树 + 写时复制。
- 生命周期淘汰（`mimir.kv_cache`）：任务边界主动回收。
- 硬件抽象（`mimir.hardware`）：CUDA/ROCm/Ascend/Cambricon/CPU 降级链。

**引擎层（vLLM v0.10.2 in-tree patch，`third_party/vllm/`）**——直接改 vLLM 源码（纯 Python，不重编 `_C`）：
- `block_pool.mimir_finish_task`：任务边界主动回收 KV（vLLM LRU 做不到）。
- `kv_cache_manager` 跨分支 CoW 复用记账。
- per-block KV-pin（lifecycle-bounded，区别于 Continuum 的 time-bounded whole-request）。
- fp8 KV 优雅降级（不支持硬件降级 bf16 而非崩溃）。
- `mimir` 调度策略 + MimirRequestQueue（自驱动自动回收）。
- `get_new_blocks` 前回收 EVICTABLE（lifecycle-aware 分配）。

详见 [`docs/VLLM_PATCH_INVENTORY.md`](VLLM_PATCH_INVENTORY.md)。

## 三·甲、诚实边界：并发吞吐下回收不占优（已验证，不回避）

我们做过 util=0.9 大池子下真·并发压测（一次提交 N 请求 / 多轮 / 请求潮三种模式）。**诚实结论**：0.9 大池子下 native 靠 vLLM preemption 能默默扛住高并发（请求都完成，仅 TTFT 退化），Mimir 的"任务结束清空 KV"在纯吞吐场景反而因重算拖累（少完成请求）——**省显存未换来吞吐**。故 Mimir 回收真价值在「长生命周期 / 显存紧张到要 OOM」的场景（见下 Phase Q 262→0），不在大池子高并发吞吐。这条边界如实写，体现工程诚实。

## 三、引擎级证据：四个决定性 A/B（used_blocks，越低越好，交替多轮场景）

在 patched v1 引擎上，原生 vLLM vs Mimir：

| 场景 | 原生 vLLM | Mimir | 说明 |
| --- | --- | --- | --- |
| 单 agent 10 轮对话（Phase M） | 74 | 0 (reclaims 239) | 每轮自动回收 |
| 3 agent 并发 6 步（Phase O） | 14 | 0 (reclaims 24) | per-task 隔离 + 回收 |
| KV 池压力 6 任务（Phase P） | 27 | 0 (reclaims 132) | lifecycle-aware 分配 |
| **工具调用并发 3agent×2轮（Phase Q，~5KB 返回）** | **262** | **0 (reclaims 42)** | **tool_offload + 回收** |

**结论**：原生 vLLM KV 持续累积，Mimir 在任务边界主动回收 + 工具外置，显存稳态为 0。Phase Q 是最大赢面（262→0）且最贴合赛题工具调用场景。

## 五、长生命周期与容错

- 分层存储：baseline 第 5 轮 OOM（存活 4/20），Mimir 三层存活全部 20 轮。
- per-block pin：agent 3 块在另一 agent 压力下 3/3 存活。
- fp8 降级：RTX 3090 不支持时降级 bf16（原会 NotImplementedError 崩溃）。
- 多模型泛化：Qwen3-1.7B + 4B 均验证 lifecycle + CoW（8B 因 3090 显存排除）。

## 六、与同实验室 Continuum 的差异化

Mimir 独有：任务边界主动回收、per-block pin（lifecycle-bounded）、CoW 记账、fp8 容错、`mimir` 调度策略、**block-class 类别感知淘汰（核心创新）**。
Continuum：tool-call-pause 的 time-bounded whole-request pin。
同源 v0.10.2，但 Mimir 在「任务边界主动回收」+「按语义角色路由 KV 淘汰」上有独占贡献。

## 六·甲、创新核心：tool-call 感知的 per-block KV 类别管理【夺冠差异化】

**为什么是创新**：业界 KV 淘汰（H2O / SnapKV / PyramidKV / KIVI）在长上下文 QA 上按 attention
权重 / 访问热度评分；agent-specific 的（IntentKV / FlowKV / MemArt）仅研究阶段、SGLang-only。
**无论文按「语义角色」标签化 KV 块并按类别路由淘汰**——而这正是 agent 与普通长上下文的关键差别：
工具返回（tool_result）是后续多步会复用的高价值数据，推理中间态（reasoning）是可丢的低价值数据。
vLLM 原生 LRU 不区分二者，盲目按访问序淘汰。

**Mimir 做法**：每个 KV 块打标签 `{system, user, reasoning, tool_result}`，按
`reasoning > user > tool_result > system` 优先级主动淘汰——优先丢 reasoning、保留 tool_result 到
任务结束再外置摘要、system 共享前缀最后。**3090 无 fp8**，KV 容量没法靠精度提升，只能靠
「谁该被淘汰」更聪明——block-class 正是 agent 场景下最大化有效 KV 容量的正解。

**证据**：
- `tests/test_block_class.py`（5 单测）：优先级序 + PINNED 跳过 + 零需求 no-op + 类别计数。
- 真实 Qwen3-4B 演示：块分布 `{system:20, reasoning:114, tool_result:99}`；`evict(57)` 只淘汰
  reasoning 57 块、tool_result/system 0 损失。
- probe 召回率佐证：class-aware 主动淘汰 342 reasoning 块、tool_result 完整存活，二次命中比 LRU 多。

## 六·乙、诚实评测：DeepSeek 真实轨迹 + LLM-judge

不用合成数据或弱模型自驱动轨迹，而用 **DeepSeek V4 Pro 产真实 agent 轨迹**作工作负载，replay 时
native/Mimir 跑同一份上下文（A/B 干净）；用 **DeepSeek-flash 作裁判**量化压缩保真度。

- 轨迹 A/B：Mimir 全程 used=0；native 在重轨迹（compare）溢出崩（peak 285），轻轨迹虽跑完但 KV 堆到 763/1235 块逼近池上限。崩溃场景匹配步 TTFT −86%。诚实 trade-off：低强度回放 Mimir offload 使前缀 hash 改变、TTFT 持平或略高——价值在显存维稳态 0 + 不崩。
- LLM-judge 保真：能跑场景压缩无损（10/10 == 10/10）；full 上下文超长直接崩，Mimir 压缩后反能答——压缩不仅是省显存，更是救任务。

## 七、可复现

```bash
git clone git@github.com:TobyMint/Mimir.git && cd Mimir
bash scripts/setup_vllm_binaries.sh   # 重建预编译二进制
source scripts/activate_env.sh
make reproduce                        # --quick CPU 模式；带 GPU 跑 phase_m/o/p/q

# 创新与真实轨迹评测（需 GPU 0 + DeepSeek API key 放 .deepseek_key）
python scripts/run_phase_blockclass.py     # 创新：block-class 类别感知淘汰
python scripts/run_recall_metric.py        # 诚实指标：probe 召回率
python scripts/gen_deepseek_traces.py      # DeepSeek V4 Pro 产真实 agent 轨迹
python scripts/run_trace_benchmark.py      # 轨迹 A/B（native 崩 vs Mimir used=0）
python scripts/run_llm_judge.py            # DeepSeek-judge 压缩保真 A/B
```
