# 三篇融合技术报告:Agent 工具调用场景下的 KV 放置管线

> 本报告记录 Mimir「三篇融合」工作:把 Continuum + LMCache + CacheGen 三篇 prior work
> 在 agent 工具调用场景下串成统一 KV 放置管线,在单卡 RTX 3090 + vLLM 0.10.2 上落地验证,
> 并给出 reload 收益的量化数字与诚实边界。
>
> 日期:2026-06-24 | 模型:Qwen3-4B-Instruct-2507 | 硬件:单卡 RTX 3090(24GB) | vLLM 0.10.2(in-tree)

---

## 1. 问题背景:agent 推理的内存三段式困境

传统 LLM 推理是「一次输入一次输出」,KV cache 用完即释。但**智能体推理**不同——多轮、带工具、长生命周期,在单卡上炸出三个具体的内存问题:

1. **工具调用暂停时,KV 该留多久?** agent 第 N 步发工具调用、暂停去执行工具,这步算出的 KV 下一秒就要复用,但传统引擎请求结束即释放 → 第 N+1 步从头重 prefill。工具执行往往几百毫秒到几秒,KV 本该留着等它。

2. **显存不够时,KV 放哪?** 单卡 24GB,KV pool 撑死 ~88K tokens(util=0.9)。多 agent 并发或单 agent 上下文膨胀,显存紧张,要么淘汰 KV(回来重算,慢),要么搬到 CPU(回来再搬回,也要快)。

3. **搬动的 KV 太大。** KV 是大浮点张量,长上下文下一个请求的 KV 能到 GB 级,跨设备搬运开销大,需要压缩。

**这三个问题对应三篇论文各解决一段**,但没有一篇把三段串成 agent 工具调用场景下的统一管线。

## 2. 任务:三篇融合,而非自造机制

| 环节 | 论文 | 贡献 | 单独的缺口 |
| --- | --- | --- | --- |
| **何时**留/放 | Continuum (arXiv 2511.02230) | 工具调用边界 + TTL 保留 + cost-benefit | offload 整个未压缩张量,搬得慢 |
| **搬去哪/怎么搬** | LMCache (arXiv 2510.09665) | GPU↔CPU↔磁盘分层 offload、layer-wise 重叠、pin/lookup | 通用搬运工,语义无感,搬原始张量 |
| **怎么压/怎么传得快** | CacheGen (arXiv 2310.07240, SIGCOMM'24) | delta+分层量化+算术编码把 KV 压成 bitstream(3.5–4.3×) | 只解决「压/传」,不解决何时留/放、搬去哪 |

**任务**:把三篇在 agent 工具调用场景下串成一条管线——agent 工具调用暂停 → Continuum 决定保留这步 KV → 显存紧时 LMCache 把它 offload → offload 的不是原始张量而是 CacheGen 编码的 bitstream → 工具回来、同 agent 下一步进来时 LMCache reload + CacheGen 解码,快于重 prefill。**三篇无人串成这条 agent 管线**——这是 Mimir 的贡献:集成 + 场景化 + 诚实归因。不自造机制、不占为己有、不夸大收益。

## 3. 方法(Action):三篇落地 + 收益验证

### 3.1 Continuum TTL 移植(何时轴)

移植 Continuum 开源代码(`Hanchenli/vllm-continuum`,同 vLLM 0.10.2 fork)进 in-tree vLLM 的 `"mimir"` 调度策略(不新增 `continuum` policy,对外仍叫 Mimir):

- `estimate_with_func.py`(新):TTL 估计器 + 工具调用 parser(扩展支持 `[TOOL:...]`/`[FINAL:...]` 格式)。
- `request.py`:加 `job_id`/`this_func_call`/`is_last_step`/`last_func_call` 字段,搭便车走 vLLM 自带 `SamplingParams.extra_args`(不改 SamplingParams 本身)。
- `request_queue.py`:`MimirRequestQueue` 吸收 program-level FCFS + pinned-job 优先。
- `scheduler.py`:pin/unpin/TTL 状态机 + 死锁兜底 + pin-on-finish。

**验证**:13 个确定性单测(parser/set_up_pin 决策/在线学习/queue 选择/pin-unpin/死锁兜底)+ 真引擎冒烟(Qwen3-4B):step1 发 `[TOOL: echo(...)]` → `pinned=1`(KV pin 住)、step2 发 `[FINAL: ...]` → `pinned=0`(末步释放、无泄露)。完整 TTL 生命周期跑通。

### 3.2 LMCache 集成(搬哪轴)

LMCache 是 vLLM 0.10.2 兼容的 CPU offload 底座。集成 `src/mimir/lmcache_compat.py` 集中处理三件让它在我们 in-tree vLLM 上可用的事:
1. **otel LoggerProvider 兜底**:lmcache 0.4.7 import 时无条件挂 LoggingHandler,otel-sdk 1.26 未 set provider 时 `emit` 崩(ProxyLogger 无 resource)——import 前先设真 provider。
2. **connector 自注册**:lmcache 0.4.7 不自带 entry point,手动 `KVConnectorFactory.register_connector`。
3. 可用性报告。

**验证**:真引擎三件套共存冒烟——`policy=MIMIR`、TTL est 就绪、`lmcache available(0.4.7 / vllm 0.10.2 / connector_registered)`。LMCache 接入达、日志确认 `Stored 4096 tokens`(agent KV 确实 offload 到 CPU)。

### 3.3 CacheGen 编解码(怎么压轴)

**关键发现(省了大量工作)**:CacheGen 的编解码器(delta+分层量化+算术编码)已随 LMCache 0.4.7 一并发布,作为 storage serde 后端(`CacheGenSerializer`,`remote_serde="cachegen"` 选用)。第三步是「配置 LMCache 用 CacheGen serde」,不是写编码器。

**验证**:`tests/test_cachegen_serde.py` 对 Qwen3-4B 真实 KV 形状(2,36,256,1024 bf16)round-trip:形状还原 + **压缩 2.88×**(合成随机张量;论文真实 KV 因 token-wise locality 达 3.5–4.3×)。模型名走 `CacheGenConfig.from_model_name` 的 AutoConfig 默认回退(Qwen3-4B: 36 层,前 10 层 K/V 用 32 bins、后续 16 bins)。

### 3.4 拿收益数字:撞墙与突破

收益的本质是「agent KV 被显存争用挤出 GPU 后,native 重 prefill(慢)、Mimir reload(快)」。要拿到对照数字,需要让「KV 被挤出」真正发生。

**撞墙(诚实记录)**:
- **压力不足 → KV 没被挤出**:初始干扰 `max_tokens=32`、跑完即释,池子没持续满,vLLM v1 APC 的懒 LRU 不触发淘汰,agent 前缀一直留 GPU 命中(收益差=0)。
- **pin-on-finish 撞 APC 假设**:Continuum 的 pin-on-finish 把 KV 留 GPU 不释放,违 vLLM v1 APC「已完成请求=释放 block 跟踪」假设,多步重用同 job/rid 时撞 `assert len(new_computed_blocks)==0`。
- **LMCache connector hash 不兼容**:LMCache 默认要 vLLM 的 `builtin` hash 函数,但 vLLM 0.10.2 无此函数,退回 Python `hash`,与 vLLM block hash 语义不兼容 → store 的 KV lookup 不命中(完全相同 prompt 也 hit=0)。换 `sha256_cbor` 后 hash 能加载,但 store 又不触发——多层接口不兼容。

**突破(换 vLLM 原生 connector)**:
vLLM 自带 `SharedStorageConnector`(debug 实现,标准 `input_ids_hash` + safetensors,store/load 自洽),绕开 LMCache hash 黑盒。验证:同 prompt 第二次 `cached=624`、`External Cache Hit!`(命中)。

**拿到收益数字**:
- **压力给够**:降 `gpu_memory_utilization` 到 0.55(KV 池缩到 ~28K token)+ 干扰 `max_tokens=1024` 持续 decode 占显存 → 池子在 agent 下一步到达时被占满 → LRU 触发 → agent 旧前缀被淘汰(不可逆)。
- **drain 干扰隔离 reload**:淘汰不可逆,drain 干扰后 agent 前缀仍不在 → step2 TTFT 是纯 reload / 纯重 prefill。
- **patch SSC debug assert**:SSC 的 `assert total_need_load == len(_requests_need_load)` 在并发压测下失败(请求加入但本步未调度)→ 放宽为只清已处理、未处理的下步重新 lookup(debug 实现健壮性,不丢数据)。

## 4. 结果(Result):量化收益数字

**实验设置**:单卡 RTX 3090, GPU3, util=0.55(KV 池 ~28K tokens / 1780 块), Qwen3-4B-Instruct-2507 bf16, vLLM 0.10.2 in-tree。agent 4 步,每步 prompt 4000 token,步间注入 20 个干扰请求(各 4000 输入 + 1024 输出)挤满池、淘汰 agent KV;drain 干扰后跑下一步,隔离 step2 的 reload/重 prefill 时间。store 到 `/dev/shm` tmpfs。

### 4.1 native vs +SSC reload 对照

| 侧 | step1 TTFT | step2 TTFT | step2 cached | step2 重算 token | total 重算 |
| --- | --- | --- | --- | --- | --- |
| **native(无 connector)** | 530ms | **530ms(全重 prefill)** | 0 | 4027 | 16108 |
| **+SSC(reload)** | 1059ms | **242ms(reload 命中)** | **4016** | **11** | 4060 |

### 4.2 收益

- **TTFT 改善**:step2 `530ms → 242ms`,**降 54%**。
- **重算 token**:step2 `4027 → 11`,**降 99.7%**(4016 token 从存储 reload 回来,免重算)。
- **多步累积**:total 重算 `16108 → 4060`,**降 75%**(native 每步全重 prefill,Mimir 只有 step1 全算 + store,后续都 reload)。

**结论**:agent 的 KV 被显存争用挤出 GPU 后,**native 每步重 prefill 4027 token(530ms),而 reload 方案从存储读回 4016 token、只重算 11 个(242ms)。reload 比重 prefill 快一倍以上**——这是「搬去哪」轴的真实、可复现的量化收益。step1 +SSC 略慢(1059 vs 530)因第一次还要 store,但 agent 多步场景 step1 只一次、后续都走 reload,收益在后续步持续。

### 4.3 CacheGen 压缩(独立指标)

Qwen3-4B 真实 KV 形状上,CacheGen 编解码器压缩 **2.88×**(合成随机张量;论文真实 KV 3.5–4.3×),round-trip 形状还原。这是「怎么压」轴的独立量化。

## 5. 诚实边界

1. **connector 用的是 vLLM 原生 SharedStorageConnector**(debug 实现,`prompt[:-1]` 精确匹配命中),**非 LMCache/CacheGen**。三篇 connector(LMCache local-CPU 原始张量 / CacheGen remote serde)在 vLLM 0.10.2 + InprocClient 设置下 hash/store 接口不兼容(详见 `docs/三篇融合评测与边界.md`)——这是版本兼容黑盒,如实记录。SSC 用标准 hash 绕开,拿到 reload 收益数字。
2. **完全相同 prompt 每步重复**(满足 SSC 精确匹配命中条件);真实 agent 多步 prompt 增长需更鲁棒的前缀部分命中 connector(LMCache/更鲁棒实现支持,但要先解决兼容)。本报告的收益数字在「agent 跨步共享固定前缀」场景下成立。
3. **store 到 `/dev/shm` tmpfs**(CPU 内存盘),模拟 LMCache 的 CPU offload 层。非真实 LMCache local-CPU 后端。
4. **patch 了 SSC debug 实现的 load-count assert**(`shared_storage_connector.py`):放宽为只清已处理、未处理的下步重新 lookup。这是 debug 实现的并发健壮性修复,不丢数据、不改命中语义。已记录在 `benchmark_results/ssc_reload_ab.json`。
5. **收益数字的归因**:这是「搬去哪」轴(LMCache 思路:KV 挤出 GPU 后 offload-reload 而非丢)的收益证据,用 vLLM 原生 connector 实现。Continuum TTL(何时)已 port 验证机制正确;CacheGen(怎么压)已验证 2.88× 压缩。三篇机制各自正确,但**在 vLLM 0.10.2 + InprocClient 设置下未能用三篇各自的 connector 串成一条数据流**——SSC 是绕开 connector 兼容黑盒、拿到 reload 收益数字的务实替代。
