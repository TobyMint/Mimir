# 三篇融合技术报告:Agent 工具调用场景下的 KV 放置管线

> 本报告记录 Mimir「三篇融合」工作:把 Continuum + LMCache + CacheGen 三篇 prior work
> 在 agent 工具调用场景下串成统一 KV 放置管线,在单卡 RTX 3090 + vLLM 0.10.2 上落地验证,
> 并给出**干扰强度扫描下的三步走收益曲线**与诚实边界。
>
> 日期:2026-06-25 | 模型:Qwen3-4B-Instruct-2507 | 硬件:单卡 RTX 3090(24GB) | vLLM 0.10.2(in-tree)

---

## 1. 问题背景:agent 推理的内存三段式困境

传统 LLM 推理是「一次输入一次输出」,KV cache 用完即释。但**智能体推理**不同——多轮、带工具、长生命周期,在单卡上炸出三个具体的内存问题:

1. **工具调用暂停时,KV 该留多久?** agent 第 N 步发工具调用、暂停去执行工具,这步算出的 KV 下一秒就要复用,但传统引擎请求结束即释放 → 第 N+1 步从头重 prefill。工具执行往往几百毫秒到几秒,KV 本该留着等它。

2. **显存不够时,KV 放哪?** 单卡 24GB,KV pool 撑死 ~88K tokens(util=0.9)。多 agent 并发或单 agent 上下文膨胀,显存紧张,要么淘汰 KV(回来重算,慢),要么搬到 CPU(回来再搬回,也要快)。

3. **搬动的 KV 太大。** KV 是大浮点张量,长上下文下一个请求的 KV 能到 GB 级,跨设备搬运开销大,需要压缩。

**这三个问题对应三篇论文各解决一段**,但没有一篇把三段串成 agent 工具调用场景下的统一管线。

## 2. 任务:三篇融合,而非自造机制

| 环节 | 论文 | 贡献 | 单独的缺口 |
| --- | --- | --- | --- |
| **何时**留/放 | Continuum (arXiv 2511.02230) | 工具调用边界 + TTL 保留 + cost-benefit | offload 整个未压缩张量,搬得慢 |
| **搬去哪/怎么搬** | LMCache (arXiv 2510.09665) | GPU↔CPU↔磁盘分层 offload、layer-wise 重叠、pin/lookup | 通用搬运工,语义无感,搬原始张量 |
| **怎么压/怎么传得快** | CacheGen (arXiv 2310.07240, SIGCOMM'24) | delta+分层量化+算术编码把 KV 压成 bitstream(3.5–4.3×) | 只解决「压/传」,不解决何时留/放、搬去哪 |

**任务**:把三篇在 agent 工具调用场景下串成一条管线——agent 工具调用暂停 → Continuum 决定保留这步 KV → 显存紧时 LMCache 把它 offload → offload 的不是原始张量而是 CacheGen 编码的 bitstream → 工具回来、同 agent 下一步进来时 LMCache reload + CacheGen 解码,快于重 prefill。**三篇无人串成这条 agent 管线**——这是 Mimir 的贡献:集成 + 场景化 + 诚实归因。不自造机制、不占为己有、不夸大收益。

## 3. 方法(Action):三篇落地 + 收益验证

### 3.1 Continuum TTL 移植(何时轴)

移植 Continuum 开源代码(`Hanchenli/vllm-continuum`,同 vLLM 0.10.2 fork)进 in-tree vLLM 的 `"mimir"` 调度策略(不新增 `continuum` policy,对外仍叫 Mimir):

- `estimate_with_func.py`(新):TTL 估计器 + 工具调用 parser(扩展支持 `[TOOL:...]`/`[FINAL:...]` 格式)。
- `request.py`:加 `job_id`/`this_func_call`/`is_last_step`/`last_func_call` 字段,搭便车走 vLLM 自带 `SamplingParams.extra_args`(不改 SamplingParams 本身)。
- `request_queue.py`:`MimirRequestQueue` 吸收 program-level FCFS + pinned-job 优先。
- `scheduler.py`:pin/unpin/TTL 状态机 + 死锁兜底 + pin-on-finish。pin 用 vLLM 原生 `block_pool.touch`(增 ref_cnt + 移出 free queue),不动 ref_cnt 手动增减(那会破坏 free_block_queue 不变量),每步 re-touch 保活。

**验证**:13 个确定性单测(parser/set_up_pin 决策/在线学习/queue 选择/pin-unpin/死锁兜底)+ 真引擎冒烟(Qwen3-4B):step1 发 `[TOOL: echo(...)]` → `pinned=1`(KV pin 住)、step2 发 `[FINAL: ...]` → `pinned=0`(末步释放、无泄露)。完整 TTL 生命周期跑通。

### 3.2 LMCache 集成(搬哪轴)

LMCache 是 vLLM 0.10.2 兼容的 CPU offload 底座。集成 `src/mimir/lmcache_compat.py` 集中处理三件让它在我们 in-tree vLLM 上可用的事:
1. **otel LoggerProvider 兜底**:lmcache 0.4.7 import 时无条件挂 LoggingHandler,otel-sdk 1.26 未 set provider 时 `emit` 崩(ProxyLogger 无 resource)——import 前先设真 provider。
2. **connector 自注册**:lmcache 0.4.7 不自带 entry point,手动 `KVConnectorFactory.register_connector`。
3. 可用性报告。

**验证**:真引擎三件套共存冒烟——`policy=MIMIR`、TTL est 就绪、`lmcache available(0.4.7 / vllm 0.10.2 / connector_registered)`。

### 3.3 CacheGen 编解码(怎么压轴)

**关键发现(省了大量工作)**:CacheGen 的编解码器(delta+分层量化+算术编码)已随 LMCache 0.4.7 一并发布,作为 storage serde 后端(`CacheGenSerializer`,`remote_serde="cachegen"` 选用)。第三步是「配置 LMCache 用 CacheGen serde」,不是写编码器。

**验证**:`tests/test_cachegen_serde.py` 对 Qwen3-4B 真实 KV 形状(2,36,256,1024 bf16)round-trip:形状还原 + **压缩 2.88×**(合成随机张量;论文真实 KV 因 token-wise locality 达 3.5–4.3×)。

### 3.4 撞墙与突破(工程过程,诚实记录)

**撞墙**:
- **LMCache connector hash 不兼容**:LMCache 默认要 vLLM 的 `builtin` hash 函数,但 vLLM 0.10.2 无此函数,退回 Python `hash`,与 vLLM block hash 语义不兼容 → store 的 KV lookup 不命中(完全相同 prompt 也 hit=0)。换 `sha256_cbor` 后 hash 能加载,但 store 又不触发——多层接口不兼容。**这是版本兼容黑盒,如实记录。**
- **pin-on-finish 撞 APC 假设**:Continuum 的 pin-on-finish 把 KV 留 GPU 不释放,违 vLLM v1 APC「已完成请求=释放 block 跟踪」假设,多步重用同 job/rid 时撞 `assert len(new_computed_blocks)==0`。手动 `ref_cnt+=1` 又破坏 `free_block_queue` 不变量(`remove() called on invalid block`)。

**突破**:
- **pin 用 vLLM 原生 `block_pool.touch`**:增 ref_cnt + 移出 free queue,不动 ref_cnt 手动增减,每步 re-touch 保活——既 pin 住 KV 又不破坏 APC 不变量(压力下不崩)。
- **换 vLLM 原生 `SharedStorageConnector`(SSC)**:debug 实现,标准 `hashlib.md5` + safetensors,store/load 自洽,绕开 LMCache hash 黑盒。并给 SSC 打前缀匹配补丁(支持 agent 多轮 prompt 增长:从长到短试 block 对齐前缀 hash)+ 容错(load 文件不存在/形状不匹配时跳过该层不崩)+ 非互斥 store/load(前缀匹配 load 时也 store 完整 prompt 供下轮匹配)+ 放宽 load-count assert(并发健壮)。
- **突破 InprocClient 同步限制**:用 in-process `add_request`+`step` 交错(8 agent 并发,各步交错提交),而非 `LLM.chat()` 串行——既造出"agent 暂停工具调用期间干扰涌入挤掉 KV"的真实争用,又保父进程可观测 `block_pool`/`scheduler` 统计(异步 server 下这些在子进程读不到)。

## 4. 结果(Result):干扰强度扫描 + 三步走

**实验设置**:单卡 RTX 3090 GPU3, util=0.55(KV 池 ~28K token), Qwen3-4B-Instruct-2507 bf16, vLLM 0.10.2 in-tree。8 agent × 10 轮 ReAct(每步 prompt 累积增长,模拟真实 agent 上下文膨胀),步间注入干扰挤显存。**扫 4 档干扰强度**(none/weak/medium/strong),固定其余参数,看 native 命中率随压力退化 + pin/native+SSC 各档表现。in-process `add_request`+`step` 交错。完整数据 `benchmark_results/interference_sweep.json`,曲线 `benchmark_results/interference_sweep.png`。

### 4.1 native 命中率随干扰压力退化(反驳「打地板」)

| 干扰档 | 干扰 token | native hit | native TTFT | native 重算 |
| --- | --- | --- | --- | --- |
| none | 0 | **90.1%** | 191ms | 15661 |
| weak | ~5K | 90.1% | 191ms | 15661 |
| medium | ~14K | 37.5% | 970ms | 99309 |
| strong | ~42K | **0.0%** | 1470ms | 158893 |

**无干扰时 native APC 自己命中 90%——证明 native 没被写坏**,0% 是真实压力退化(干扰负载 ~42K 超过 KV 池 ~28K,LRU 把 agent KV 全挤掉)非造假。曲线 90%→90%→37.5%→0% 是 native 随争用压力的真实退化。这是对「native 0% 是不是刻意打地板」质疑的直接回应:**不是,native 在无压/弱压下本就高命中,压力越大退化越狠**。

### 4.2 pin 增量随压力增长(pin 只在 native 保不住时发挥)

| 干扰档 | native hit | pin hit | pin 增量 | pin TTFT |
| --- | --- | --- | --- | --- |
| weak | 90.1% | 90.1% | **+0%** | 192ms |
| medium | 37.5% | 78.6% | **+41.1%** | 430ms |
| strong | 0.0% | 78.3% | **+78.3%** | 442ms |

**pin 增量 = native 保不住的比例**。弱压 native 已 90%(APC 够用),pin 保活多余、零增量(没活干);重压 native 0%,pin 保活把命中率拉到 78%、TTFT 降 70%(strong 442ms vs native 1470ms,KV 留 GPU 免 reload)。这是 pin 收益的诚实触发边界:**pin 不创造命中,只在 native 会丢 KV 时保住它**——压力越大越值。

### 4.3 native+SSC:reload 恢复命中率但 TTFT 被 reload 拖累

| 干扰档 | native hit | native+SSC hit | SSC 增量 | SSC TTFT |
| --- | --- | --- | --- | --- |
| weak | 90.1% | 77.6% | **−12.5%(负收益)** | 688ms |
| medium | 37.5% | 85.6% | **+48.1%** | 1129ms |
| strong | 0.0% | 78.4% | **+78.4%** | 1457ms |

SSC reload 从 `/dev/shm` 恢复命中率(重压 +78%,和 pin 同档),但 reload 读盘+memcpy 开销让 TTFT **几乎不降**(strong 1457ms ≈ native 1470ms)。弱压下 SSC 反而**负收益**(APC 够用,reload 是纯开销,hit 77.6% < native 90.1%)。SSC 用 TTFT 换命中率:medium SSC 85.6% > pin 78.6%(恢复更多 KV),但 TTFT 1129ms >> pin 430ms。

### 4.4 pin+SSC 冲突:SSC 在 pin 工作时空转(诊断)

pin+SSC 组合下,SSC 前缀匹配命中 611 次但 **Inject KV=0**(诊断 `scripts/diag_pinsc.py`,2 agent × 3 round strong 干扰)——pin 保活让 APC `num_computed_tokens` 已等于 `matched`,SSC `get_num_new_matched_tokens` 返回 `matched − computed = 0`,不进 load 路径。pin 和 SSC 作用在同一批已 store 的 KV 上,功能重叠;agent 多轮**新增长**的 token(上轮输出+tool_result)谁也没 store 过、谁也覆盖不了。故三步走第三步用 **native+SSC**(SSC 代替 pin)而非 pin+SSC——后者 SSC 空转 + store 开销(strong/pinsc 988s,远超 pin 269s)。

### 4.5 三步走总结(strong 档,8 agent × 10 轮)

| 步 | 配置 | hit | TTFT | 重算 | total |
| --- | --- | --- | --- | --- | --- |
| 1 native | 无优化 | 0% | 1470ms | 158893 | 224s |
| 2 +pin | GPU 保活(Continuum TTL) | 78.3% | 442ms | 34477 | 269s |
| 3 +native+SSC | 存储 reload(vLLM SSC) | 78.4% | 1457ms | 15534 | 426s |

**读数**:
- **命中率**:pin 和 SSC 都把 0% 拉到 ~78%(KV 不丢则不丢,丢则保/reload 回来)。这是单调的——三步走「逐步提升」在命中率/重算量上成立。
- **TTFT(非单调,诚实)**:pin 降 70%(KV 留 GPU 免 reload,442ms);SSC 没降(reload 读盘+memcpy 开销抵消命中,1457ms ≈ native)。
- **total(非单调,诚实)**:pin 269s 反超 native 224s——保活占显存降吞吐,是 pin 降 TTFT 的代价;SSC 426s 最慢(store 每步写盘开销)。

**诚实结论**:**pin 是更优的单进程方案**(同命中率、TTFT 大降)。SSC 的价值不在单进程内(pin 已覆盖同一批 KV),而在**跨进程/持久化**(pin 只保活进程内 KV,SSC store 到 `/dev/shm` 可跨进程/重启复用)——这是 SSC 相对 pin 的唯一独占场景,单进程 benchmark 体现不出。三步走不是「步步全优」,而是 **pin 保活(快)+ SSC 持久化(持久)各擅一长**,命中率单调提升、TTFT/total 各有代价——如实呈现。

## 5. 诚实边界

1. **LMCache/CacheGen connector 在 vLLM 0.10.2 + InprocClient 设置下 hash/store 接口不兼容**(详见 `docs/三篇融合评测与边界.md`)——版本兼容黑盒。三步走第 3 步用 vLLM 原生 `SharedStorageConnector`(标准 `hashlib.md5` + safetensors)绕开,store 到 `/dev/shm` tmpfs(模拟 LMCache 的 CPU offload 层,非真实 LMCache local-CPU 后端)。
2. **SSC 打了前缀匹配补丁**(`shared_storage_connector.py`):支持 agent 多轮 prompt 增长(从长到短试 block 对齐前缀 hash)+ 容错(load 文件不存在/形状不匹配跳过)+ 非互斥 store/load + 放宽 load-count assert。这是 debug 实现的并发健壮性修复,不丢数据、不改命中语义。
3. **pin+SSC 在单进程内功能重叠**:** SSC load 在 pin 已保活时不触发(`get_num_new_matched_tokens` 返回 0),SSC 退化为纯 store 开销。这是 pin(GPU 保活)与 SSC(存储 reload)作用在同一批 KV 上的结构性冲突,不是 bug——它界定了 SSC 的独占价值在跨进程/持久化。
4. **pin 的 TTL 用 Continuum simplified 释放版**(固定 2.0s,非论文 cost-benefit P(τ,f) CDF 估算)。cost-benefit 估算列为可选增强;release 版足够证明机制有效。
5. **收益规模**:单卡 3090 util=0.55(KV 池 ~28K)下,pin 把 strong 档命中率从 0% 拉到 78%、TTFT 降 70%。Continuum 论文多卡 B200 场景收益规模可能更大——如实报数,不夸大。
6. **Continuum TTL 是 credited prior work**(arXiv 2511.02230),移植并入 Mimir,不占为己有。我们的贡献:三机制组合定位、SSC 前缀匹配补丁、干扰强度扫描证伪「打地板」、pin/SSC 收益触发边界的诚实量化。

## 7. 下一步(如继续)

1. **SSC 接 CacheGen serde**:目前 SSC 存原始 safetensors,接 CacheGen 编码能再省存储/带宽(已验证 2.88× 压缩,需接进 SSC 数据流)。
2. **LMCache connector 兼容深修**:让 LMCache local-CPU 后端在 vLLM 0.10.2 + InprocClient 下真正串通(hash/store/lookup 接口适配),替代 SSC 的 tmpfs 模拟。
3. **跨进程 SSC 验证独占价值**:当前单进程内 pin 已覆盖 SSC 的命中场景;起多进程/重启复用 `/dev/shm` store,正面证明 SSC 的跨进程持久化收益(pin 给不了)。
4. **Continuum cost-benefit TTL**:用论文的 P(τ,f) CDF 估算替换固定 2.0s TTL,按工具历史执行时间动态调 pin 时长。
