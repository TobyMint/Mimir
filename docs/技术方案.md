# 技术方案

> 本文档阐述 Mimir 的整体技术方案：要解决的问题、核心思路、各优化方向的设计与预期效果。
> 状态：block-class 类别感知淘汰 + 工具外置/压缩 + 分层 + CoW + DeepSeek 评估**已实现并验证**；
> 工具调用边界 TTL 保留（移植 Continuum）+ LMCache 分层 offload 集成为**规划中**（见 §3.7）。

## 1. 问题分析：智能体推理的内存特征

智能体推理相较传统线性生成（如 vLLM 面向的批式解码），在内存使用上呈现以下显著特征，
每一项都对应潜在的优化空间：

| 特征 | 描述 | 带来的问题 | Mimir 应对模块 |
| --- | --- | --- | --- |
| KV Cache 持续累积 | 多轮交互不重置上下文 | 显存随轮次线性增长直至 OOM | `mimir.kv_cache` |
| 上下文高度冗余 | system prompt / 工具描述逐轮重复 | 重复计算与重复存储 | `mimir.context` |
| 推理路径分支 | 多候选决策树 / Tree-of-Thought | 分支导致 KV 指数膨胀 | `mimir.branch` |
| 工具中间数据 | 工具返回大 JSON / 长文本 | 全量进入 KV，挤占显存 | `mimir.tools` |
| 生命周期长 | 单任务可执行数十~数百步 | 资源长期占用、工具调用暂停时 KV 被挤掉致重 prefill | `mimir.kv_cache` / `mimir.tiered` |
| 异构硬件 | CUDA / DTK / CANN 多后端 | 抽象与适配成本高 | `mimir.hardware` |

## 2. 总体目标

在**保证任务成功率基本不下降**的前提下：

1. **降低显存峰值占用**（核心指标，对应评分 40 分）
2. **降低端到端推理延迟**（核心指标，对应评分 40 分）
3. **保持方法通用性**，可适配不同模型规模（Qwen 1.5B/7B、MiniCPM3-4B 等）与不同推理框架
4. **具备可复现性**，统一的 Benchmark 与相同硬件配置

## 3. 核心技术路线

### 3.1 KV Cache 生命周期管理（`mimir.kv_cache`）

- **前缀复用（Prefix Caching）**：识别多轮对话 / Agent Loop 中稳定不变的上下文前缀（system
  prompt、few-shot、工具 schema），仅计算一次并在后续轮次复用其 KV。
- **类别感知淘汰（block-class）**：为 KV 块打语义类别标签，按 `reasoning > user > tool_result > system`
  优先级主动淘汰，而非仅按访问热度（LRU）——把宝贵显存留给高复用的 tool_result / system（详见 §3.7）。
- **工具调用边界 TTL 保留（规划中，移植自 Continuum）**：agent 暂停调工具时，给其 KV 挂 TTL 保留，
  同一 job 下一步进来时 KV 还在、免重 prefill；TTL 到期或显存紧时按类别降级（GPU→CPU offload→压缩→丢），
  用 LMCache 当搬运底座（详见 §3.7）。

> 注：早期版本的「任务边界主动回收 / used_blocks→0」机制已删除——系任务结束后瞬时计数偷换
> （推理时该占仍占、回收重算拖慢服务），已彻底移除。其位置由上述 TTL 保留 + 类别感知淘汰
> （sound 机制）取代。

**预期效果**：长对话场景下重复前缀的 prefill 时间趋近于零；工具调用后下一步免重 prefill；
KV 压力下高价值块（tool_result/system）存活、低价值块（reasoning）先淘汰/降级。

### 3.2 分支推理内存共享（`mimir.branch`）

- **Tree-based KV with Copy-on-Write**：将分支推理组织为树结构，共享公共前缀的 KV 块。
- **CoW 语义**：仅当某分支真正修改（重新生成）对应 token 的 KV 时才物理拷贝，其余共享。
- **分支裁剪与回收**：被剪枝的分支 KV 立即回收。

**预期效果**：N 路分支的显存从 O(N·L) 降至接近 O(L + N·Δ)，Δ 为分支差异长度。

### 3.3 Prompt 与上下文压缩（`mimir.context`）

- **冗余检测与去重**：对 system prompt、工具描述等静态内容做哈希指纹，跨轮复用。
- **语义压缩**：对历史对话 / 工具返回进行摘要或结构化精简（可选，保真度可控）。
- **静态/动态分离**：静态内容复用前缀 KV，动态内容增量处理。

**预期效果**：减少 prefill 计算量与 KV 占用，同时不损失关键信息。

### 3.4 工具调用数据优化（`mimir.tools`）

- **结构化外置存储**：工具返回的大对象（长 JSON / 表格 / 图像）存入外部 store，仅在 KV 中
  保留引用与摘要。
- **按需加载（lazy materialization）**：仅在后续步骤真正引用时才展开。
- **截断与回填策略**：超长返回按策略截断，需要时回填。

**预期效果**：工具中间数据不再整体进入 KV Cache，显存占用大幅下降。

### 3.5 分层内存与异构存储（`mimir.tiered`）

- **三级分层**：GPU 显存（热）→ 主存（温）→ 外部存储（冷），类似 CPU L1/L2/L3。
- **冷热分离与动态迁移**：基于访问频率与生命周期预测，自动在层级间迁移 KV。
- **按需加载 / 重计算权衡**：冷数据可按需 reload 或按需重算，权衡访存与算力。

**预期效果**：突破单卡显存限制，支撑超长生命周期任务。

### 3.6 异构 AI 加轾件支持（`mimir.hardware`）

- **设备抽象层**：统一 `Device` 抽象，屏蔽 CUDA / DTK（昇腾） / CANN 差异。
- **后端探测与降级**：运行时探测可用后端，无 GPU 时降级到 CPU / offload。
- **国产化算力适配**：针对国产 NPU（昇腾 / 寒武纪等）的算子与内存模型适配。

**预期效果**：在 CUDA 与国产化算力环境下均可运行，提升兼容性（对应评分 20 分）。

### 3.7 创新核心：工具调用边界触发的、按语义分层的 KV 放置

**问题**：智能体长生命周期推理中，KV 管理有三个相互正交的决策维度，业界各有侧重却无人贯通：

| 维度 | 代表工作 | 决策内容 | 缺口 |
| --- | --- | --- | --- |
| **何时**释放 | Continuum（arXiv 2511.02230） | 工具调用暂停时挂 TTL 保留 KV | 整块 pin/evict，不区分块价值 |
| **哪个 agent**的前缀该淘汰 | KVFlow（arXiv 2507.07400） | 按 steps-to-execution 调度感知淘汰 | 把非固定前缀一刀切为"后缀"全淘汰 |
| **哪种语义角色**该先淘汰 | **Mimir block-class** | 按块语义类别（system/user/reasoning/tool_result）路由 | — |

三者并非竞争而是**互补正交轴**。Continuum 把整个请求 KV 当一整块 pin 或 evict——要么全留 GPU（占显存挤别人）、要么全丢（下一步全量重 prefill 或全量 reload）。KVFlow 把 agent 的"变化后缀"统一打成最高淘汰优先级，不区分其中 reasoning（可丢）与 tool_result（高价值、后续多步复用）。LMCache（arXiv 2510.09665）是 GPU↔CPU↔压缩的通用搬运底座（reload 已比重 prefill 快），但语义无感、按 LRU/热度搬——其论文提到一家 agent 公司来要"按内容定位/压缩/转移 KV"的 API，LMCache 给了 API 却没给 agent 感知的策略。

**Mimir 创新**：在**工具调用边界**这个触发点上，把刚结束那步的 KV 块**按语义角色分流到不同存储层**，在 TTL 时间窗内决策，用 LMCache 当搬运底座，block-class 当路由策略（粘合剂）：

- **system / 共享前缀**：已在 GPU 且跨 agent 共享 → 不动（或 LMCache pin 供跨 agent 复用）
- **tool_result（高价值，后续多步复用）**：TTL 窗内 pin GPU；显存紧 → offload 到 CPU（LMCache，下一步 reload 而非重 prefill）
- **reasoning（低价值，出完答案就丢）**：LMCache compress 压缩存，或直接丢——不浪费 GPU/CPU
- **user / history（中）**：offload 到 CPU，resume 时 reload
- TTL 到期不整块 evict，按类别降级（GPU → CPU → 压缩 → 丢）

**三方归因（诚实）**：Continuum 贡献"何时"——工具调用边界 + TTL 时间窗 + cost-benefit 框架（已 clone `Hanchenli/vllm-continuum`，同 vLLM 0.10.2，计划移植其 TTL 机制）；LMCache 贡献"搬去哪/怎么搬"——分层搬运、layer-wise 与计算重叠、reload>recompute 翻转、pin/compress 控制台 API（计划集成为 offload 底座）；**block-class 贡献"哪类块走哪层"**——语义路由策略，是让融合不退化为 naive 拼接的粘合剂。block-class 不作为"压过 Continuum 的夺冠点"，而是融合的必要组成。对外这套机制统称 **Mimir**。

**为什么是 3090/Agent 架构强关联**：3090（Ampere SM_86）无 fp8 硬件，vLLM 的 fp8 KV 路径在此被拒（`flash_attn_supports_fp8()=False`），单卡上 KV 容量无法靠精度提升，只能靠更聪明的"哪类块该留/该搬/该丢"——agent 场景里 tool_result 的高复用率正是 offload-then-reload 胜过每步重 prefill 的关键（LMCache 在多轮 QA 上 15× 吞吐即此模式）。

**block-class 已实现部分**（作为分层路由的语义标签基础，已落地）：给每个 KV 块打语义类别标签 `{system, user, reasoning, tool_result}`，按 `reasoning > user > tool_result > system` 优先级主动淘汰（`block_pool.mimir_class_aware_evict(need)`，容量紧张时先触发，把宝贵显存留给 tool_result/system 而非 reasoning）。标签经 `engine_vllm_v1.chat_full` 用 tokenizer 把 prompt 按消息角色对齐到块边界算出，经 `engine_core` → `Request` → `block_pool.cache_full_blocks` 注入。

> **状态**：block-class 标签 + 类别感知淘汰已实现并验证（见下）；Continuum TTL 移植、LMCache 集成、完整分层放置策略为**规划中**（见 `VLLM_PATCH_INVENTORY.md` 与 ROADMAP）。本节描述的是目标方案，未实现部分如实标注，不充已实现。

**验证（block-class 已落地部分）**：`tests/test_block_class.py`（5 个确定性单测覆盖优先级序 / ref_cnt 守卫 / 零需求 no-op / 类别计数）；`scripts/run_phase_blockclass.py` 在真实 Qwen3-4B 上展示 `evict(57) 只淘汰 reasoning 57 块，tool_result/system 0 损失`；`scripts/run_recall_metric.py` 用「tool-call probe 召回率」佐证类别感知让 probe 块更长寿。详见 §5 与 `VLLM_PATCH_INVENTORY.md` Phase BC。

## 4. 系统集成：两层优化 + vLLM 内核级 patch

Mimir 的优化分两层，通过统一入口 `MemoryManager` 串起：

### 4.1 外部层（请求侧变换，`src/mimir/`）

在把请求喂给 vLLM 之前改写：
- **上下文压缩**（`mimir.context`）：摘要旧对话轮次 + 精简工具描述（保真度可调：LOSSLESS/BALANCED/AGGRESSIVE）——真实减少进 KV 的 prompt。
- **工具数据外置**（`mimir.tools`）：工具返回的大对象（~5KB JSON）存入 `ToolDataStore`，上下文仅放轻量引用 + 摘要；按需 `materialize()` 取回——真实减少必需 KV。
- **分层存储**（`mimir.tiered`）：GPU/HOST/DISK 三层冷热分离，自动 demote/promote——避免 OOM。
- **分支 CoW 记账**（`mimir.branch`）：分支树 + 写时复制，统计共享节省。
- **硬件抽象**（`mimir.hardware`）：CUDA/ROCm/Ascend/Cambricon/CPU 降级链。

> 注：早期版本的「生命周期淘汰/任务边界主动回收」（mimir.kv_cache 的 finish_task）已删除——其 used_blocks→0 系任务结束后瞬时计数偷换（推理时该占仍占、回收重算拖慢服务），已彻底移除。Mimir 的显存收益回归工具外置/压缩真实减少必需 KV、分层避免 OOM。

### 4.2 引擎层（vLLM v0.10.2 in-tree patch，`third_party/vllm/`）

直接修改 vLLM 源码（纯 Python，不重编 `_C`），把优化下沉到引擎内核：
- `kv_cache_manager.get_computed_blocks`：跨分支 CoW 复用记账。
- `arg_utils` fp8 优雅降级（不支持硬件降级 bf16 而非崩溃）。
- **【创新】`block_pool.mimir_block_class` + `mimir_class_aware_evict()`**：per-block 语义类别标签
  + 按类别优先级淘汰（详见 §3.7）——3090 无 fp8 下靠「谁该被淘汰」更聪明提升有效 KV 容量。
- `output_processor`（Phase R）：v1 TTFT 可观测回填。

> 注：原 lifecycle 主动回收 / per-block pin / mimir 调度策略自驱动回收已删除（used_blocks→0 系偷换概念）。

详见 [`VLLM_PATCH_INVENTORY.md`](VLLM_PATCH_INVENTORY.md)。

### 4.3 统一入口

```python
mm = MemoryManager(features=["context_compress", "tool_offload", "tiered"])
eng = VLLMEngineV1(cfg, device=0)  # 内核 block-class 标签 + CoW 记账
r = mm.run_turn_with_engine(eng, case, task_id="t1")
```

## 5. Benchmark 与评估方案

### 5.1 Agent-Loop Benchmark（核心评测）

不是预构造消息，而是让 LLM 真正驱动多步任务循环：
LLM 生成 → 解析 tool_call → 执行 mock 工具 → 结果喂回 → LLM 再生成 → ...

3 个任务（research_kv_cache / compare_frameworks / multi_step_estimate），每个最多 8-10 步。

A/B 对照：同一任务/种子/GPU：
- **native**：fcfs 策略，工具结果全量进 KV，无类别感知、无 TTL 保留
- **Mimir**：mimir 策略 + block-class 类别感知淘汰 + 工具外置/压缩（已落地）；
  工具调用边界 TTL 保留 + LMCache 分层 offload 为规划中增强（见 §3.7）

指标：每步 `used_blocks` / `ttft_ms` / `e2e_latency` + `peak_used_blocks`。

```bash
python scripts/run_agent_benchmark.py --tool-kb 5 --max-tokens 64
```

### 5.2 内核层创新（block-class 类别淘汰 + CoW，不靠回收刷计数器）

回收机制已删除（used_blocks→0 系偷换概念，见测试报告 §5.0）。内核层保留的真创新：

| 内核机制 | 说明 | 实测 |
| --- | --- | --- |
| block-class 类别淘汰（Phase BC，创新） | 块打语义标签按优先级淘汰 | evict(57) 全 reasoning、tool_result/system 0 损失；5 单测 |
| CoW 跨分支复用（Phase D） | 分支共享前缀记账 | 4 分支 9 次复用 |
| v1 TTFT 可观测（Phase R） | 回填 RequestMetrics | 每请求 TTFT/prefill 可读 |
| fp8 优雅降级（Phase F） | 不支持硬件降级 bf16 | 3090 上不崩 |

### 5.3 task-call probe 召回率（佐证 block-class 价值，诚实指标）

构造 agent 场景：长上下文含「工具调用 probe」（key 埋在 tool_result）+ 大量 reasoning 干扰。
KV 压力下对比 native(LRU) vs Mimir(class-aware 优先淘汰 reasoning)：

| 路径 | 第二次命中 token | 说明 |
| --- | --- | --- |
| native(LRU) | 1088 | 盲淘汰，probe 块部分被挤 |
| Mimir(class-aware) | 1136 | 主动淘汰 342 reasoning 块、tool_result 完整存活 |

「正确性」代理：probe 二次命中越多=agent「越没失忆」。对比 LongBench 均分，这是 agent 语义下
最诚实的正确性代理。策略正确性由 `tests/test_block_class.py` 5 单测保证。

### 5.4 DeepSeek V4 Pro 真实轨迹 + LLM-judge 评估

**真实轨迹**：用 DeepSeek V4 Pro 充当 agent 跑 mock 工具，产 3 条真实多步轨迹（9-10 步、
工具数据 21-64KB），作为 benchmark 工作负载——相比弱模型（Qwen3-4B）自驱动的轨迹更真实，
replay 时 native/Mimir 跑**同一份上下文**，A/B 干净。

| 任务 | native | Mimir | native 峰值 used | Mimir 峰值 used | 能否跑完 |
| --- | --- | --- | --- | --- | --- |
| compare_frameworks | 崩 | 3 步 | 285 | **0** | native 溢出崩，Mimir 跑完 |
| multi_step_estimate | 7 步 | 7 步 | 763 | **0** | 都跑完，Mimir 显存稳态 0 |
| research_kv_cache | 10 步 | 10 步 | 1235 | **0** | 都跑完，Mimir 显存稳态 0 |

Mimir 全程 `used_blocks=0`；native 在重轨迹下溢出崩溃，轻轨迹虽跑完但 KV 堆到 763/1235 块逼近 1780 池上限。崩溃场景匹配步 TTFT −86%（compare 316 vs 43 ms）。诚实 trade-off：低强度回放下 Mimir offload 使前缀 hash 改变、APC 命中率下降，TTFT 与 native 持平或略高——显存换重算，价值在显存维稳态 0 + 不崩。

**LLM-judge 保真 A/B**：用 DeepSeek-flash 作裁判，对 Qwen3-4B 在 full vs Mimir 压缩上下文
（−91%~−97%）下回答分别评分（相对参考答案 0-10）：

| 任务 | full 上下文 | Mimir 压缩上下文 | 说明 |
| --- | --- | --- | --- |
| compare_frameworks | 10/10 ✅ | 10/10 ✅ | 压缩无损 |
| multi_step_estimate | 0/10（65K 字符 > max_model_len 崩）| 0/10 | 压缩后能跑但未给出估算 |
| research_kv_cache | 0/10（62K 字符崩）| 见 judge | full 直接溢出崩 |
| **平均** | 3.33 | 5.0 | **压缩无损**·full 上下文超长直接崩，Mimir 压缩后反能答 |

结论：上下文压缩对能跑的场景无损（compare_frameworks 10/10 == 10/10）；对 full 上下文
超 `max_model_len` 直接崩的场景，Mimir 压缩使其可运行——**压缩不仅是省显存，更是救任务**。

## 6. 风险与权衡

| 风险 | 应对 |
| --- | --- |
| 上下文压缩可能损失信息 | 保真度可配置；task-success delta=0 验证不降质量 |
| 分层存储带来额外访存延迟 | 冷热分级；热数据常驻显存 |
| vLLM 内部改动维护成本 | 纯 Python patch（不重编 `_C`）；mimir_* 方法均为 additive |
