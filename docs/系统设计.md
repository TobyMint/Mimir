# 系统设计

> 本文档描述 Mimir 的系统架构、模块划分、核心数据结构与接口设计。
> 反映实际实现（Phase A-Q），详见 [`VLLM_PATCH_INVENTORY.md`](VLLM_PATCH_INVENTORY.md)。

## 1. 架构总览

Mimir 分两层优化，通过统一 `MemoryManager` 入口串起：

```
  Agent Workload
       │
       ▼
  ┌─────────────────────────────────────────┐
  │     外部层（src/mimir/）请求侧变换        │
  │  context_compress → tool_offload → ...  │
  └────────────────┬────────────────────────┘
                   │ 变换后的请求
                   ▼
  ┌─────────────────────────────────────────┐
  │  引擎层（third_party/vllm/）vLLM patch   │
  │  block-class 标签 + CoW 记账 + TTFT 回填 │
  └────────────────┬────────────────────────┘
                   │
                   ▼
              vLLM v1 Engine (InprocClient)
              Qwen3-4B on single RTX 3090
```

### 关键设计决策

- **v1 InprocClient**（`VLLM_USE_V1=1 VLLM_ENABLE_V1_MULTIPROCESSING=0`）：引擎在主进程内，父进程可直接遍历 `llm.llm_engine.engine_core.engine_core.scheduler.kv_cache_manager.block_pool` 读取 KV 块使用。
- **纯 Python patch**（不重编 `_C`）：预编译 `.abi3.so` 来自 wheel，与 torch 2.8.0+cu128 ABI 匹配，symlink 进 `third_party/vllm/vllm/`。Python 改动改即生效。
- **普通目录（非 submodule）**：`third_party/vllm/` 是 tracked 普通目录，patch 直接改 `vllm/v1/core/...`。

## 2. 模块划分

### 2.1 外部层（`src/mimir/`）

| 模块 | 文件 | 职责 | 关键类 |
| --- | --- | --- | --- |
| 上下文压缩 | `context/compressor.py` | 启发式压缩（截断/结构摘要），Fidelity 三档 | `ContextCompressor` |
| 语义压缩 | `context/semantic.py` | LLM 驱动摘要（用模型自身），带缓存 | `LLMSemanticCompressor` |
| 工具外置 | `tools/offload.py` | 大返回存 `ToolDataStore`，上下文留引用 | `ToolDataStore`, `ToolDataRef` |
| 分层存储 | `tiered/store.py` | GPU/HOST/DISK 三层 LRU + 迁移 | `TieredStore` |
| 分支 CoW | `branch/cow.py` | 分支树 + 朴素 vs CoW KV 记账 | `BranchTree` |
| 多任务协调 | `multitask.py` | 单卡跨任务 KV 隔离 + 共享前缀 | `MultiTaskCoordinator` |
| 硬件抽象 | `hardware/device.py` | CUDA/ROCm/Ascend/Cambricon/CPU 降级链 | `DeviceBackend` |
| 引擎适配(v0) | `engine_vllm.py` | v0 单进程引擎封装 + KV block 查询 | `VLLMEngine` |
| 引擎适配(v1) | `engine_vllm_v1.py` | v1 InprocClient + block-class 注入 + 批量 | `VLLMEngineV1` |
| 统一管线 | `manager.py` | 特性开关编排 + `run_turn_with_engine` | `MemoryManager` |

> 注：原「生命周期淘汰」模块（`kv_cache/lifecycle.py` 的 task-boundary 主动回收）已停用——其 used_blocks→0 系偷换概念（推理时该占仍占、回收重算拖慢服务）。`LifecycleEvictor`/`PureLRUEvictor` 保留为模拟对照类，不再作为卖点。显存收益回归工具外置/压缩真实减少必需 KV、分层避免 OOM。

### 2.2 引擎层 in-tree patch（`third_party/vllm/vllm/v1/`）

| 文件 | patch | 行为 |
| --- | --- | --- |
| `core/block_pool.py` | `mimir_block_class` + `mimir_class_aware_evict()`（按 reasoning>user>tool_result>system 优先级淘汰）；`mimir_block_task`（CoW 记账） | block-class 类别淘汰（创新）；CoW 复用记账 |
| `core/sched/scheduler.py` | `get_mimir_stats()` 块级统计导出（used/total/CoW/block_class） | 观测 |
| `core/kv_cache_manager.py` | `get_computed_blocks` CoW 记账 | 跨分支复用计数 `mimir_cow_reuses` |
| `engine/core.py` | `req.mimir_task_id`（CoW 用）+ `req.mimir_block_classes` 注入 | 标签流入 block_pool |
| `engine/arg_utils.py` | fp8 优雅降级 | 不支持硬件 warn + 回退 bf16 |
| `engine/output_processor.py`（Phase R） | TTFT 回填 RequestMetrics | v1 每请求 TTFT/prefill 可观测 |

> 原 `mimir_finish_task`/`mimir_pin_blocks`/`mimir_reclaim_evictable`/`mimir_used_blocks`/mimir 策略自驱动回收已删除（used_blocks→0 偷换概念）。

## 3. 核心数据结构

### 3.1 block-class 语义标签（引擎层，创新）

```
BlockClass（block_id -> class）:
  system       — system prompt（共享前缀，最忌淘汰，留到最后）
  tool_result  — 工具返回（高价值，agent 后续多步复用）
  user         — 用户消息
  reasoning    — assistant 推理中间态（低价值，优先淘汰）
优先级：reasoning > user > tool_result > system（reasoning 最先淘汰）
```

`mimir_class_aware_evict(need)`：按类别优先级主动淘汰缓存块（仅 ref_cnt==0 的可淘汰块）。对比 vLLM 原生 LRU（盲按访问序）——block-class 按语义价值淘汰。

### 3.2 外部层管线

```
PipelineResult:
  case: WorkloadCase      — 变换后的请求
  steps: list[PipelineStep]  — 每步的名称/启用/指标

MemoryManager.apply(case):
  prefix_cache(记账) → semantic_compress → context_compress → tool_offload → tiered(记账) → ...
```

### 3.3 Agent Loop（Benchmark）

```
AgentLoop:
  messages = [system + user]
  loop:
    LLM.generate(messages) → text
    parse_tool_call(text) → (tool_name, arg) or [FINAL: answer]
    if tool: execute mock_tool → result
      if offload: store.put(result), context = ref
      else: context = result (full)
    messages += [assistant: text, tool: context]
  per-step: record ttft / new_prefill / used_blocks
```

## 4. 关键流程

### 4.1 单步 Agent Loop（Mimir 全管线）

1. `MemoryManager.apply(case)`：压缩历史 + 外置工具返回（真实减少进 KV 的 prompt）。
2. `VLLMEngineV1.set_current_task(task_id)`：注入 task_id（CoW 记账用）。
3. `VLLMEngineV1.chat_full(messages)`：注入 block-class 标签 → vLLM 生成。
4. `mimir_stats()` 读 used_blocks / CoW / block_class（观测，不靠回收刷 0）。
5. native 若工具返回全量进上下文 → 多步后上下文超 `max_model_len` 崩溃；Mimir 外置压缩后不超长、跑完。

### 4.2 Native（无优化）

1. 工具返回全量进上下文。
2. 多步后上下文膨胀 → 超过 `max_model_len` → **崩溃**。

## 6. 扩展性与非目标

- **扩展**：新优化方向 → 外部层加模块 (`src/mimir/<name>/`) + `MemoryManager.SUPPORTED_FEATURES`；引擎层加 `mimir_*` 方法 + `get_mimir_stats()` 导出。
- **非目标**：不重新实现通用推理引擎；vLLM patch 仅纯 Python（不重编 `_C`）；不做 llama.cpp 后端（赛题允许二选一）。
