## 2. 模块划分

每个模块对应赛题的一个优化方向，职责单一、可独立启用：

| 模块 | 职责 | 核心类 |
| --- | --- | --- |
| `mimir.kv_cache` | KV 复用、淘汰、生命周期管理 | `KVBlock`, `KVCacheManager`, `EvictionPolicy` |
| `mimir.branch` | 分支推理 KV 共享与 CoW | `BranchTree`, `CoWBlock` |
| `mimir.context` | Prompt / 上下文压缩与去重 | `ContextCompressor`, `PrefixFingerprint` |
| `mimir.tools` | 工具中间数据外置与按需加载 | `ToolDataStore`, `LazyRef` |
| `mimir.tiered` | 分层内存与冷热迁移 | `TieredStore`, `MigrationPolicy` |
| `mimir.hardware` | 设备抽象与异构后端 | `Device`, `Backend` |
| `mimir.core` | 统一编排与策略引擎 | `MemoryManager`, `BlockAllocator`, `PolicyEngine` |

## 3. 核心数据结构

### 3.1 KV Block（统一块抽象）

将 KV Cache 划分为逻辑块（如 PagedAttention 的 block 思想），每个块带元数据：

```
KVBlock:
    block_id      : int          # 全局唯一块 ID
    seq_id        : str          # 所属序列/会话
    token_range   : (start, end) # 覆盖的 token 区间
    ref_count     : int          # 引用计数（共享/CoW 用）
    lifecycle     : Enum         # ACTIVE / EVICTABLE / PINNED
    device_tier   : Enum         # GPU / HOST / DISK
    hash          : bytes        # 内容指纹（前缀复用匹配用）
```

### 3.2 Branch Tree（分支推理）

```
BranchNode:
    node_id       : int
    parent        : BranchNode | None
    blocks        : List[KVBlock]   # 该节点独有（非共享父节点）的块
    children      : List[BranchNode]
    pruned        : bool
```

公共前缀块由多个 BranchNode 共享（通过 `ref_count`），写入时触发 CoW。

### 3.3 统一接口：MemoryManager

```python
from mimir import MemoryManager

mm = MemoryManager(
    backend="vllm",
    features=["prefix_cache", "branch_cow", "tool_offload"],
    device="cuda",
)

# Agent loop 中透明使用
with mm.step(seq_id="turn_3", parent="turn_2"):
    mm.add_prefix(system_prompt)         # 自动复用 / 压缩
    mm.attach_tool_result(tool="search", data=large_json)  # 外置，不入 KV
    ...
```

## 4. 关键流程

### 4.1 多轮对话（前缀复用）

1. 计算本轮上下文指纹 → 命中已缓存前缀 → 跳过 prefill
2. 未命中 → 计算 KV 并注册为可复用
3. 历史块按生命周期标记，必要时淘汰

### 4.2 工具调用

1. 工具返回大数据 → 写入 `ToolDataStore`，得到 `LazyRef`
2. 仅 `LazyRef` + 摘要进入 KV
3. 后续步骤若引用 → 按需 materialize

### 4.3 分支推理（CoW）

1. 分支节点继承父块引用（`ref_count++`）
2. 重新生成某 token → CoW，物理拷贝该块
3. 分支被剪枝 → 释放独有块，`ref_count--`

## 5. 配置与可观测性

- 配置驱动：各特性可通过 `features=[...]` 开关，便于消融实验
- 指标埋点：块命中/淘汰/迁移计数、显存峰值、各阶段耗时
- 日志与 trace：便于测试报告与归因

## 6. 扩展性与非目标

- **扩展性**：新优化方向可作为新模块插入，经 `PolicyEngine` 编排
- **非目标**：不重新实现通用推理引擎，而是作为 vLLM / llama.cpp 的内存管理层扩展
