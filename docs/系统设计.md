# 系统设计

> 本文档描述 Mimir 的系统架构、模块划分、核心数据结构与接口设计。
> 反映实际实现（Phase A-Q），详见 [`VLLM_PATCH_INVENTORY.md`](VLLM_PATCH_INVENTORY.md)。

## 1. 架构总览

Mimir 分两层优化，通过统一 `MemoryManager` 入口串起：

```
  Agent Workload
       │
       ▼
  ┌─────────────────────────────────────────┐
  │     外部层（src/mimir/）请求侧变换        │
  │  context_compress → tool_offload → ...  │
  └────────────────┬────────────────────────┘
                   │ 变换后的请求
                   ▼
  ┌─────────────────────────────────────────┐
  │  引擎层（third_party/vllm/）vLLM patch   │
  │  mimir 策略 → auto-reclaim → KV=0      │
  └────────────────┬────────────────────────┘
                   │
                   ▼
              vLLM v1 Engine (InprocClient)
              Qwen3-4B on single RTX 3090
```

### 关键设计决策

- **v1 InprocClient**（`VLLM_USE_V1=1 VLLM_ENABLE_V1_MULTIPROCESSING=0`）：引擎在主进程内，父进程可直接遍历 `llm.llm_engine.engine_core.engine_core.scheduler.kv_cache_manager.block_pool` 读取 KV 块使用。
- **纯 Python patch**（不重编 `_C`）：预编译 `.abi3.so` 来自 wheel，与 torch 2.8.0+cu128 ABI 匹配，symlink 进 `third_party/vllm/vllm/`。Python 改动改即生效。
- **普通目录（非 submodule）**：`third_party/vllm/` 是 tracked 普通目录，patch 直接改 `vllm/v1/core/...`。

## 3. 核心数据结构

### 3.1 KV 块生命周期（引擎层）

```
BlockLifecycle:
  ACTIVE    — 属于进行中任务，不可回收
  EVICTABLE — 任务已结束，可立即回收
  PINNED    — 常驻（system prefix / agent 暂停 pin）
```

`mimir_finish_task(task_id)`：遍历 `mimir_block_task`，把该任务块标记/物理释放。仅释放 `ref_cnt==0` 且非 PINNED 的块（mirror vLLM 原有 guard）。

`mimir_reclaim_evictable()`：扫描所有 EVICTABLE 块，物理释放。在 `get_new_blocks` 前调用（lifecycle-aware 分配）。

### 3.2 外部层管线

```
PipelineResult:
  case: WorkloadCase      — 变换后的请求
  steps: list[PipelineStep]  — 每步的名称/启用/指标

MemoryManager.apply(case):
  prefix_cache(记账) → semantic_compress → context_compress → tool_offload → tiered(记账) → ...
```

### 3.3 Agent Loop（Benchmark）

```
AgentLoop:
  messages = [system + user]
  loop:
    LLM.generate(messages) → text
    parse_tool_call(text) → (tool_name, arg) or [FINAL: answer]
    if tool: execute mock_tool → result
      if offload: store.put(result), context = ref
      else: context = result (full)
    messages += [assistant: text, tool: context]
  per-step: record used_blocks / ttft / e2e
```

## 4. 关键流程

### 4.1 单步 Agent Loop（Mimir 全管线）

1. `MemoryManager.apply(case)`：压缩历史 + 外置工具返回。
2. `VLLMEngineV1.set_current_task(task_id)`：注入 `engine_core._mimir_current_task`。
3. `VLLMEngineV1.chat(messages)`：vLLM 生成。
4. 请求完成 → `_free_blocks` 自动调 `mimir_finish_task(task_id)` → KV 块释放。
5. `mimir_stats()` 读 `used_blocks=0` / `lifecycle_reclaims++`。

### 4.2 Native（无优化）

1. 工具返回全量进上下文。
2. 多步后上下文膨胀 → 超过 `max_model_len` → **崩溃**。
3. 即使不崩溃，KV 块持续累积无回收。

## 6. 扩展性与非目标

- **扩展**：新优化方向 → 外部层加模块 (`src/mimir/<name>/`) + `MemoryManager.SUPPORTED_FEATURES`；引擎层加 `mimir_*` 方法 + `get_mimir_stats()` 导出。
- **非目标**：不重新实现通用推理引擎；vLLM patch 仅纯 Python（不重编 `_C`）；不做 llama.cpp 后端（赛题允许二选一）。
