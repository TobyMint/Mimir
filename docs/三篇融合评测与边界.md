# 三篇融合:评测、实测与诚实边界

> 本文记录 Continuum + LMCache + CacheGen 三篇融合在**单卡 RTX 3090 + Qwen3-4B + vLLM 0.10.2 in-tree**上的
> 评测过程、实测数据与**诚实的收益边界**。动机:用户要求"结果一定要有说服力"——
> 而最大的说服力来自诚实——把哪些收益拿到了、哪些拿不到、为什么拿不到,如实写清。
>
> 日期:2026-06-25。**端到端收益已复现**(突破 InprocClient 同步限制),详见 §3。完整数字见 `docs/三篇融合技术报告.md` §4。

## 1. 三篇各自的评测口径(调研结论)

| 论文 | 负载 | 干净指标 | 收益触发条件 |
| --- | --- | --- | --- |
| **Continuum** (arXiv 2511.02230) | 真实 BFCL/SWE-Bench 轨迹(40–500 并发 program,6–11 轮,长尾 tool time) | P90/P95 job duration、重 prefill 次数 | **显存争用**(并发 agent × 上下文 > KV 池) |
| **LMCache** (arXiv 2510.09665) | emulated 多轮 QA(非 agent) | TTFT、cache hit ratio、loading 带宽 | 显存稀缺才需 offload;32 Gbps 下要 >256K token 才赢 prefill |
| **CacheGen** (arXiv 2310.07240) | LongChat/TriviaQA(非 agent) | TTFT、KV size、SLO violation | **网络是瓶颈**(低带宽/远程存储);本地高带宽 PCIe 收益主要在省 CPU 内存 |

**关键诚实结论**:三篇收益的触发条件**不完全一致** —— Continuum/LMCache 靠显存争用,
CacheGen 靠低带宽。不能指望"并发争用自动给三篇都带来收益"。

真实 agent 负载数据点(AgentBench/τ-bench/SWE-Bench/Continuum trace):
- agent 任务 **5–50 轮**;Continuum 实测 SWE-Bench 10.9 轮、BFCL 6.3 轮
- tool 执行时间**长尾严重**(最慢 10% 占总延迟 52–94%) —— 这正是 TTL 收益的核心场景
- 上下文:Continuum 实测 70K–93K tokens/program

## 3. 端到端收益 benchmark —— 干扰强度扫描 + 三步走(已突破 InprocClient 限制)

### 3.1 突破:in-process `add_request`+`step` 交错

之前认为 InprocClient 同步模式做不到端到端收益量化(`LLM.chat` 串行、多线程并发 `llm.chat` 触发
`Forward context is not set` 崩、同步批量 gap 期间无"持续涌入")。**突破**:用 in-process
`add_request`+`step` 交错——8 agent 并发,各步交错提交请求,step 循环跑完一轮再注入干扰。
既造出"agent 暂停工具调用期间干扰涌入挤掉 KV"的真实争用(持续涌入),又保父进程可观测
`block_pool`/`scheduler` 统计(异步 server 下这些在子进程读不到)。这是拿到三步走收益数字的
关键工程突破——**不牺牲可观测性也能造出争用**。

### 3.2 干扰强度扫描:反驳「native 0% 是刻意打地板」

固定 util=0.55(KV 池 ~28K token)/agents=8/rounds=10/ctx=2000,扫 4 档干扰强度,看 native
命中率随压力退化:

| 干扰档 | 干扰 token | native hit | native TTFT | native 重算 |
| --- | --- | --- | --- | --- |
| none | 0 | **90.1%** | 191ms | 15661 |
| weak | ~5K | 90.1% | 191ms | 15661 |
| medium | ~14K | 37.5% | 970ms | 99309 |
| strong | ~42K | **0.0%** | 1470ms | 158893 |

**无压 native 90% 命中——证明 native 没被写坏**,0% 是真实压力退化(干扰 ~42K 超 KV 池 ~28K,
LRU 把 agent KV 全挤掉)。曲线 90%→90%→37.5%→0% 是 native 随争用压力的真实退化,非打地板。
这是对「native 0% 是不是刻意打地板」质疑的直接回应。

### 3.3 三步走(strong 档,8 agent × 10 轮)

| 步 | 配置 | hit | TTFT | 重算 | total |
| --- | --- | --- | --- | --- | --- |
| 1 native | 无优化 | 0% | 1470ms | 158893 | 224s |
| 2 +pin | GPU 保活(Continuum TTL,`touch` 保活) | 78.3% | 442ms | 34477 | 269s |
| 3 +native+SSC | 存储 reload(vLLM SSC + 前缀匹配补丁) | 78.4% | 1457ms | 15534 | 426s |

- **pin 增量随压力**:weak +0%(APC 够用,pin 无活干)→ medium +41.1% → strong +78.3%。
  pin 不创造命中,只在 native 会丢 KV 时保住它——压力越大越值。
- **SSC 弱压负收益**(77.6% < native 90.1%,reload 纯开销)→ 重压 +78.4%(恢复命中率但 TTFT
  没降,reload 读盘+memcpy 开销抵消命中)。SSC 用 TTFT 换命中率。
- **pin+SSC 冲突**:SSC load 在 pin 保活时**完全不触发**(Inject KV=0,诊断 `scripts/diag_pinsc.py`
  证实:前缀匹配命中 611 次但 0 次 inject)。pin 保活让 APC `num_computed_tokens` 已等于 `matched`,
  SSC `get_num_new_matched_tokens` 返回 0。pin/SSC 作用同一批已 store KV,功能重叠——故第三步用
  **native+SSC**(SSC 代替 pin)而非 pin+SSC(后者 SSC 空转 + store 开销,strong/pinsc 988s)。

**诚实非单调**:命中率/重算**单调**提升(0→78→78%、158K→34K→15K);TTFT/total **不单调**
(pin 降 TTFT 70% 但 total 反超 native 224→269s,保活占显存降吞吐;SSC TTFT 没降 1457≈native 1470,
reload 开销;SSC total 最慢 426s,store 每步写盘)。**pin 是更优单进程方案**(同命中率 + TTFT 大降);
**SSC 独占价值在跨进程/持久化**(pin 只保活进程内 KV,SSC store 到 `/dev/shm` 可跨进程/重启复用),
单进程 benchmark 体现不出——这是 SSC 的边界,如实标注。

### 3.4 CacheGen 收益单独看(非显存争用驱动)

CacheGen 收益靠**低带宽/远程存储**,不是显存争用。单卡本地 CPU offload 走 PCIe(~32–64 GB/s),
压缩的收益主要在**省 CPU 内存与 KV size**,不在 TTFT(LMCache 论文亦如此结论)。
`test_cachegen_serde.py` 已实测 Qwen3-4B KV 压缩 2.88×(合成;真实 KV 3.5–4.3×)。
要在 TTFT 上体现,需要**远程/低带宽存储后端**(LMCache `remote_serde="cachegen"` + 远端),
单卡本地设置体现不出来——这是 CacheGen 论文本身的边界。

## 4. 结论(诚实)

1. **三篇机制全部正确落地并有验证**(Continuum pin 机制、LMCache 接入、CacheGen 2.88× 压缩)。
2. **端到端收益已复现**(突破之前认为的 InprocClient 同步限制):用 `add_request`+`step` 交错
   造出争用,干扰强度扫描拿到三步走曲线——native 90%→0% 真实退化(没打地板)、pin 增量随压力
   0→+78%、SSC 重压 +78%。
3. **诚实非单调**:命中率/重算单调提升,TTFT/total 各有代价(pin 降 TTFT 但降吞吐、SSC reload
   拖累 TTFT)。**pin 是更优单进程方案;SSC 独占价值在跨进程/持久化**——单进程内 pin 已覆盖 SSC
   的命中场景,SSC 的 reload 在 pin 保活时不触发(诊断证实),这是结构性边界非 bug。
4. **LMCache/CacheGen connector 在 vLLM 0.10.2 + InprocClient 兼容黑盒未解**:三步走第 3 步用 vLLM
   原生 `SharedStorageConnector`(标准 `hashlib.md5` + safetensors)绕开 hash 黑盒,store 到 `/dev/shm`
   tmpfs 模拟 CPU offload 层。真实 LMCache local-CPU 后端需深修 hash/store/lookup 接口(下一步)。
5. 这是**设置突破 + 诚实边界**(透明如实报告):TTL pin 在冒烟 + 干扰扫描里机制正确性与收益都已证;
   收益数字可信复现(native 退化曲线 + pin/SSC 增量随压力),非单点不可复现。

**如何进一步放大收益(诚实路径,如要继续)**:
- **跨进程 SSC 验证独占价值**:起多进程/重启复用 `/dev/shm` store,正面证明 SSC 的跨进程持久化
  收益(pin 给不了),把 SSC 从"单进程内被 pin 覆盖"推到"独占场景显优势"。
- **接 CacheGen serde 进 SSC**:SSC 存原始 safetensors → CacheGen 编码,省存储/带宽(已验证 2.88×)。
- **LMCache connector 兼容深修**:让三篇各自 connector 在 vLLM 0.10.2 + InprocClient 下真正串通。
- **Continuum cost-benefit TTL**:用论文 P(τ,f) CDF 估算替换固定 2.0s TTL,动态调 pin 时长。
