# 三篇融合:评测、实测与诚实边界

> 本文记录 Continuum + LMCache + CacheGen 三篇融合在**单卡 RTX 3090 + Qwen3-4B + vLLM 0.10.2 in-tree**上的
> 评测过程、实测数据与**诚实的收益边界**。动机:用户要求"结果一定要有说服力"——
> 而最大的说服力来自诚实——把哪些收益拿到了、哪些拿不到、为什么拿不到,如实写清。

## 1. 三篇各自的评测口径(调研结论)

| 论文 | 负载 | 干净指标 | 收益触发条件 |
| --- | --- | --- | --- |
| **Continuum** (arXiv 2511.02230) | 真实 BFCL/SWE-Bench 轨迹(40–500 并发 program,6–11 轮,长尾 tool time) | P90/P95 job duration、重 prefill 次数 | **显存争用**(并发 agent × 上下文 > KV 池) |
| **LMCache** (arXiv 2510.09665) | emulated 多轮 QA(非 agent) | TTFT、cache hit ratio、loading 带宽 | 显存稀缺才需 offload;32 Gbps 下要 >256K token 才赢 prefill |
| **CacheGen** (arXiv 2310.07240) | LongChat/TriviaQA(非 agent) | TTFT、KV size、SLO violation | **网络是瓶颈**(低带宽/远程存储);本地高带宽 PCIe 收益主要在省 CPU 内存 |

**关键诚实结论**:三篇收益的触发条件**不完全一致** —— Continuum/LMCache 靠显存争用,
CacheGen 靠低带宽。不能指望"并发争用自动给三篇都带来收益"。

真实 agent 负载数据点(AgentBench/τ-bench/SWE-Bench/Continuum trace):
- agent 任务 **5–50 轮**;Continuum 实测 SWE-Bench 10.9 轮、BFCL 6.3 轮
- tool 执行时间**长尾严重**(最慢 10% 占总延迟 52–94%) —— 这正是 TTL 收益的核心场景
- 上下文:Continuum 实测 70K–93K tokens/program

## 3. 端到端收益 benchmark —— 诚实的边界(核心)

### 3.1 我们想要的:并发多轮 agent 下 TTL/offload 的收益

Continuum TTL 的收益场景:**agent A 暂停调工具(数秒~数十秒)→ 期间持续有新请求涌入 →
A 的 KV 被挤掉 → A 回来要重 prefill → TTL pin 保住 A 的 KV 免重算。** 这需要(Continuum 论文的设置):
- **异步 server**(`vllm serve`,持续接收在线请求)
- **持续请求涌入**(agent 暂停期间池子被新请求填满)

### 3.2 我们的设置为什么做不到

我们用 **vLLM v1 InprocClient 同步**模式(为了父进程可观测 `block_pool`/`scheduler` 统计):
- `LLM.chat()` 走引擎锁**串行化** —— 一次调用阻塞到这批请求全部完成。
- 多线程并发 `llm.chat` 尝试(交错 agent):触发 `Forward context is not set` 崩溃
  (vLLM v1 InprocClient 不支持多线程并发提交)。
- 同步批量(N agent 同 batch 进出):gap 期间**全部暂停**、无"持续涌入"→ TTL pin 在
  冒烟(单步)里证明机制正确,但端到端无重叠场景。

**这是设置限制,非机制缺陷。** Continuum 论文本身也是用 `vllm serve` 异步 server 跑的。

### 3.3 实测数据(InprocClient 同步,单卡 3090)

**run_target_interferer.py**(单 agent 多步 + 干扰批,机制级验证)—— 5 步、ctx 2000、
干扰 24×3000≈72K tokens(逼近 88K KV 池):

| 侧 | max_pinned | step1 TTFT | step2 TTFT | step2 cached | hit_ratio |
| --- | --- | --- | --- | --- | --- |
| native (fcfs) | **0** | 302ms | 20ms | 2064 | 0.80 |
| mimir (TTL) | **1** ✓ | 333ms | 28ms | 2064 | 0.80 |

读数:
- **TTL pin 正确触发**:mimir `max_pinned=1`(强制工具调用 prompt 下 step1 发 `[TOOL:...]`、
  estimator 解析、pin KV),native=0。**机制正确性得证。**
- 但 native 与 mimir 的 step2 几乎一样(均 ~20ms、cached 2064 全命中):**vLLM v1 的 APC
  (ref-counted LRU-tree)在干扰批"用完即释"下没顶掉 target 跨步共享前缀**——APC 已足够,
  TTL pin 无增量收益。
- 根因:InprocClient 同步模式下,干扰批是一次性提交、跑完即释放,无法"持续涌入"把
  target 前缀挤掉。这正是 §3.2 的设置限制。

**run_concurrent_fusion.py**(批量并发基线)—— 20 agent × 5 轮 × 1500 ctx(显存争用):

| 侧 | mean TTFT | hit_ratio | total new_prefill |
| --- | --- | --- | --- |
| native (fcfs) | 1209ms | 0.50 | 31500 |

20×5×1500 下显存确实争用(hit 从无争用的 0.74 跌到 0.50、TTFT 暴涨到 1209ms、
31.5K token 被重算)——**这说明争用场景构造出来了**。但同步批量 gap 期间无重叠,TTL 与
native 在这组数据下无差异(TTL 需要重叠才能起作用,而 InprocClient 给不了重叠)。

### 3.4 CacheGen 的收益单独看(非显存争用驱动)

CacheGen 收益靠**低带宽/远程存储**,不是显存争用。单卡本地 CPU offload 走 PCIe(~32–64 GB/s),
压缩的收益主要在**省 CPU 内存与 KV size**,不在 TTFT(LMCache 论文亦如此结论)。
`test_cachegen_serde.py` 已实测 Qwen3-4B KV 压缩 2.88×(合成;真实 KV 3.5–4.3×)。
要在 TTFT 上体现,需要**远程/低带宽存储后端**(LMCache `remote_serde="cachegen"` + 远端),
单卡本地设置体现不出来——这是 CacheGen 论文本身的边界。

## 4. 结论(诚实)

1. **三篇机制全部正确落地并有验证**(Continuum pin 机制、LMCache 接入、CacheGen 2.88× 压缩)。
2. **端到端收益的量化受 InprocClient 同步设置限制,无法可信复现**:Continuum TTL 的在线交错
   收益需异步 server + 持续涌入,我们同步设置给不了;CacheGen 的 TTFT 收益需低带宽/远端,
   单卡本地给不了。
3. 这是**设置限制(透明如实报告),非机制缺陷**:TTL pin 在冒烟 + target+interferer(max_pinned)
   里机制正确性已证;收益数字需换异步 server 跑,超出本项目 InprocClient 可观测设置范围。

**如何获得收益数字(诚实路径,如要继续)**:
- 切到 vLLM **异步 server**(`vllm serve` + Continuum fork 的 TTL patch + LMCache + CacheGen
  serde),用 BFCL/SWE-Bench 真实轨迹 + 持续 Poisson 涌入,跑 native vs full 的 P90/P95 job
  duration —— 这正是 Continuum 论文的设置,能复现其 ~2× 收益。
- 单卡 3090 跑得动(Continuum 论文也用单 A100/B200 跑),代价是**放弃 InprocClient 可观测性**
  (异步 server 下 block_pool 在子进程,父进程读不到统计)——需另搭可观测通道。
